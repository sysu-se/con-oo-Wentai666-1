# 项目结构与完成思路

## 一、项目概览

这是一个基于 Svelte + TailwindCSS 的数独游戏项目。核心思路是将“领域对象”与 Svelte 的响应式 UI 分离，领域代码负责数独规则与状态管理，UI 负责呈现和交互。

## 二、目录结构

- `src/`
  - `App.svelte`：顶层入口，负责页面布局、全局 modal、游戏开始和胜利检测。
  - `main.js`：挂载 Svelte 应用。
  - `components/`：UI 组件集合。
    - `Board/`：数独棋盘与单元格渲染。
    - `Controls/`：操作栏与虚拟键盘。
    - `Header/`：顶部按钮和下拉菜单。
    - `Modal/`：弹窗管理及具体弹窗类型。
  - `domain/`：业务逻辑层。
    - `Sudoku.js`：数独规则、验证、克隆、序列化等核心对象。
    - `Game.js`：游戏生命周期、Undo/Redo、订阅机制。
    - `GameDomainAdapter.js`：领域对象与 UI 状态的适配器。
    - `index.js`：统一导出领域对象。
  - `stores/`：Svelte store 适配器，用于将领域状态暴露给组件。
    - `gameStore.js`：将 `Game` / `Sudoku` 与 Svelte 响应式 store 连接。

- `tests/hw1/`：单元测试目录，验证数独契约、基本规则、克隆、撤销/重做、序列化等功能。
- `rollup.config.js`：构建配置。
- `package.json`：依赖和脚本。

## 三、主要模块说明

### 1. `src/domain/Sudoku.js`

这是数独的核心领域对象：
- 持有当前 `grid` 和 `initialGrid`。
- 提供 `guess(move)` 执行用户输入，并禁止修改初始数字。
- 提供 `isValid()`、`isComplete()` 校验规则。
- 提供 `clone()`，用于创建独立快照支持 Undo/Redo。
- 提供 `getInvalidCells()`，用于返回当前冲突单元格集合。

### 2. `src/domain/Game.js`

`Game` 负责游戏过程控制：
- 保存当前 `Sudoku` 实例。
- 管理 `past` / `future` 两个栈，实现撤销与重做。
- 执行 `guess()` 时先保存快照，再调用 `Sudoku.guess()`。
- 提供 `undo()` / `redo()` / `reset()` / `isWon()`。
- 通过 `subscribe(callback)` 通知 UI 更新。

### 3. `src/domain/GameDomainAdapter.js`

这个适配器是领域层与 UI 层之间的桥梁：
- 初始化 `Game` 实例。
- 将 UI 触发的操作（guess、undo、redo、reset）转为领域调用。
- 统一输出当前游戏状态对象，供组件渲染使用。

### 4. `src/stores/gameStore.js`

这个文件将领域对象包装成 Svelte store：
- 创建 `grid`、`invalidCells`、`won`、`canUndo`、`canRedo` 等可订阅 store。
- 通过 `gameInstance.subscribe(updateState)` 将领域状态同步到 store。
- 对外提供 `initialize()`、`guess()`、`undo()`、`redo()`、`reset()` 等方法。

## 四、UI 与数据流

### 1. 启动流程

- `src/main.js` 挂载 `App.svelte`。
- `App.svelte` 负责加载欢迎对话框、监听胜利状态，并渲染 `Header`、`Board`、`Controls`、`Modal`。
- 游戏初始状态通过 `GameDomainAdapter` / `gameStore` 创建 `Game` 实例并初始化 `Sudoku`。

### 2. 用户输入流程

- 用户点击棋盘某个单元格或按虚拟键盘。
- 组件触发 `guess({ row, col, value })`。
- 该调用进入 `Game.guess()`：保存当前快照、清空 redo、执行 `Sudoku.guess()`。
- `Game` 通过订阅者更新当前状态。
- `store` 接收状态后，Svelte 组件自动重新渲染。

### 3. Undo / Redo 流程

- 点击撤销时，`Game.undo()` 将当前状态推入 `future`，从 `past` 中恢复前一个状态。
- 点击重做时，`Game.redo()` 将当前状态推入 `past`，从 `future` 中恢复。
- UI 中的按钮状态由 `canUndo` / `canRedo` 驱动。

## 五、我的完成思路

### 1. 逐层分离职责

- 将“数独规则”与“游戏流程”拆成 `Sudoku` 和 `Game`。
- `Sudoku` 只关心网格合法性与修改规则，不处理历史。
- `Game` 只关心历史、撤销/重做、胜利状态。
- `GameDomainAdapter` 和 `gameStore` 负责将领域逻辑暴露给 UI。

### 2. 使用快照保证状态安全

- 使用 `Sudoku.clone()` 深拷贝状态。
- 每次 `guess()` 前保存快照，避免对象共享导致撤销失败。
- `past` / `future` 均保存独立的 `Sudoku` 实例。

### 3. 用订阅模式连接领域和界面

- `Game.subscribe()` 让 UI 不必直接轮询。
- 当领域状态变化时，立即同步到 Svelte store。
- 这样既保留了领域对象的纯粹性，又兼顾 Svelte 的响应式机制。

### 4. 关注可测试性

- 通过 `tests/hw1/` 验证核心业务逻辑。
- 推荐先学习 `Sudoku` 和 `Game` 的测试用例，再对照实现理解设计意图。

## 六、建议的学习顺序

1. 先读 `src/domain/Sudoku.js`，理解数独校验、克隆和冲突检测。
2. 再读 `src/domain/Game.js`，理解撤销/重做和状态订阅机制。
3. 看 `src/stores/gameStore.js`，理解领域对象如何映射到 Svelte store。
4. 最后读 `src/components/Board/index.svelte`、`src/components/Controls/index.svelte` 和 `src/components/Modal/index.svelte`，理解 UI 如何消费状态。

## 七、运行与测试

- `npm run dev`：启动开发模式。
- `npm run build`：构建生产包。
- `npm test`：运行所有测试。
- `npm run test:game`：单独运行游戏 undo/redo 测试。

---

> 这个项目的核心价值在于“领域对象 + UI 响应式适配”的分层设计。理解 `Sudoku` 和 `Game` 的职责分离，是把业务逻辑和界面结构拆开的关键。
